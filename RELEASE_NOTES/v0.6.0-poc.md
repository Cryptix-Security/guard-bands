# v0.6.0-poc

Cryptographic role-separation release for Guard Bands.

v0.5.0 separated untrusted data and trusted instructions into two services.
This release separates their **cryptographic capabilities**: with a shared
HMAC secret, whoever can verify a band can also forge one — a compromised
verifier becomes a signer. As of v0.6.0 the two-channel architecture uses
Ed25519, so the control plane holds only the public key and a fully
compromised control plane still cannot fabricate data-plane provenance.

## Ed25519 in the core crypto

- Resolver keys may now be raw bytes (HMAC-SHA256, unchanged), Ed25519
  private keys (sign + verify), or Ed25519 public keys (**verification-only**
  — cryptographically unable to mint bands).
- The algorithm tag (`GBv1-HMAC-SHA256` / `GBv1-Ed25519`) is derived from the
  resolved key's type and bound into the authenticated payload, so a band can
  never verify under a different algorithm than it was signed with
  (cross-algorithm confusion fails closed). Signature-length gates are
  per-algorithm (32 vs 64 bytes).
- Keypair helpers included; `make dual-channel-keys` emits a fresh pair in
  env-file form. No wire-format change; existing HMAC bands verify unchanged.

## Two-channel hardening

- **No development fallback keys.** Both planes resolve key material through
  the pluggable secret provider (`SECRETS_BACKEND=env|aws|vault`) and fail
  closed at startup if it is missing or malformed. The old repo-known
  `dual-channel-dev-secret` is gone entirely.
- **True role separation.** The data plane holds the Ed25519 private key; the
  control plane holds only the public key. The demo now ends by showing the
  control plane failing to forge:
  `wrap_content -> ValueError: verification-only and cannot sign`.
- Shared constants moved to the package root so each service imports only
  what it needs and each container carries only its own key.

## Two-channel Compose deployment

`deploy/docker-compose.dual-channel.yml` runs the planes as two containers on
separate ports and **deliberately disjoint networks** — they never talk to
each other; the signature is the only bridge of trust:

```bash
make dual-channel-keys > .env.dual-channel
docker compose -f deploy/docker-compose.dual-channel.yml \
  --env-file .env.dual-channel up --build
```

Split key delivery per container, fail-closed startup at both the compose and
service layers, healthchecks, restart policies, and resource limits.
Verified end to end: a band minted in the data-plane container verifies in
the control-plane container across the network split.

## Security status at release

- Local pytest suite passes: 86 tests (14 new).
- GitHub Actions CI passed on the feature commits; CodeQL runs on push.
- Branch protection on `main`: required status checks, no force pushes.

This remains a proof-of-concept release, now with cryptographic role
separation suitable for split-trust evaluation. Remaining known work:
distributed replay ledger for multi-replica deployments, automated key
rotation, and application-specific authorization policy.
