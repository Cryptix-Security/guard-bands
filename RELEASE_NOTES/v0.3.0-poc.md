# v0.3.0-poc

Cryptographic hardening release for Guard Bands.

This release strengthens the core boundary mechanism so that every field carried
in a Guard Band marker is authenticated, adds self-contained freshness, and
records provenance for each band.

**Breaking change:** the wire format changed. Bands minted by v0.2.0-poc no
longer verify, and the redundant SHA-256 marker hash (`h`) has been removed.

New marker format:

```text
⟪INERT:START:v:1:r:b64url(nonce):iat:issued_at:exp:expires_at⟫
[untrusted content]
⟪INERT:END:mac:b64(mac):kid:keyid:iss:b64url(issuer)⟫
```

Notable changes:

- Full marker-metadata authentication. The MAC now binds a domain-separated
  algorithm tag (`GBv1-HMAC-SHA256`), protocol version, key id, and issuer,
  preventing downgrade and metadata tampering before the protocol ever gains a
  second version or algorithm.
- Authenticated freshness. Issued/expiry timestamps (`iat`/`exp`) are bound into
  the MAC and enforced only after the MAC proves them genuine, so a tampered
  expiry cannot extend a band's lifetime. Bands now fail closed on staleness
  independent of the replay ledger. TTL is configurable via
  `GUARD_BAND_TTL_SECONDS` (default 900s).
- Provenance and a narrower signing oracle. Each band records an authenticated
  issuer; the `/wrap` endpoint stamps the SSO principal and rejects context that
  contradicts the authenticated user.
- Simplification. The redundant SHA-256 content hash is gone — the MAC is the
  sole integrity guarantee — which also removes an info-leaking error path.
- Property-based fuzzing. Hypothesis tests assert the hand-rolled parser never
  crashes and never false-accepts, alongside new tests for metadata, issuer, and
  expiry tampering.
- Documentation refreshed to the new format across the README, paper,
  architecture, serialization, API, and demo docs.
- Removed the stale `Guard-Bands-Paper.pdf` snapshot; `Guard-Bands-Paper.md` is
  now the single source of truth.

Security status at release:

- Local pytest suite passes: 49 tests (was 39).
- GitHub Actions CI and CodeQL run on push and are expected to pass.

This remains a proof-of-concept release. The replay ledger is still single-node
(in-memory or SQLite). Production use still requires environment-specific key
management, authorization design, shared replay storage, TLS, audit retention,
monitoring, and operational review.
